Public Standards Repository Guide

A formal document typography tear sheet for Markdown converted from Google Docs.

Author: Standards Maintainer Date: May 7, 2026 Status: Draft for review Classification: Public

1. Document Purpose

This tear sheet demonstrates a restrained standards style for formal documents exported from Google Docs to Markdown, then rendered as a static page or PDF. The design is intentionally quiet, operator-readable, and suitable for public repository documentation.

Body copy supports strong emphasis, emphasis, links to reference material, and inline snippets such as publication-check or docs/publishing.md.

1.1 Scope

The stylesheet is intended for standards, guides, policy notes, operating procedures, and exported planning documents. It should not look like a chat transcript, dashboard, or marketing page.

1.1.1 Repository Placement

Document assets belong in assets/typography/. Filenames should stay lowercase and descriptive, except for established names such as README.md.

Heading Level Four

Lower heading levels should remain useful for converted Google Docs outlines without becoming visually noisy.

Heading Level Five

Use small uppercase treatment for deep sections, appendices, and checklist subsections.

Heading Level Six

Reserve level six for rare converted-document structures.

2. Lists and Structure

Lists should preserve the rhythm of Google Docs while becoming cleaner and more compact in the rendered standards format.

  • Use clear, specific filenames.
  • Keep public examples sanitized.
    • Use placeholders for infrastructure names.
    • Avoid private account names, internal hostnames, or screenshots with local state.
  • Prefer durable standards over temporary notes.
  1. Draft the standard in Google Docs.
  2. Export or convert the document to Markdown.
  3. Review the Markdown for public repository readiness.
    1. Run a filename audit.
    2. Run a publication safety review.

3. Quotes and Callouts

Formal documents often quote policy language, review notes, or short excerpts from source material. The blockquote should read as supporting context.

Standards are most useful when they make the correct path easier to repeat.

Note

Use notes for clarifications, assumptions, and non-blocking review context.

Warning

Do not publish secrets, private identifiers, real infrastructure names, or operational details that belong in a private runbook.

4. Tables, Captions, and Code

Tables should remain readable in print and PDF output, with top-aligned content for multi-line Markdown cells.

Monospace blocks are useful for short config examples, command excerpts, and generated Markdown snippets.

- Replace private names with placeholders.
